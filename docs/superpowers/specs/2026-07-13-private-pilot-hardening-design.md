# Northstar 私测客户版加固设计

## 目标

把现有 Northstar RAG 从“可演示原型”升级为可交给首批 3-10 位客户使用的受控私测版本。本轮不增加模拟看板、多 Agent、账号体系或付费功能，只解决客户首次使用会直接遇到的移动端、访问控制和调用滥用问题。

## 当前验证结论

- 线上 `/readyz` 正常，知识库包含 515 个片段。
- 一次真实标准提问耗时约 9.6 秒，返回 5 个来源，回答链路可用。
- 桌面端视觉可以用于客户演示。
- 390px 手机视口出现横向溢出，快捷问题、说明文字和发送按钮被裁切。
- 登录页仅在前端检查访问码长度，没有向后端验证；错误码要到第一次提问才被发现。
- 后端无条件把 `fb300` 加入允许列表，修改 Render 的 `APP_API_KEY` 后旧码仍可能有效。
- 公网接口缺少基础频率和问题长度保护，存在消耗模型额度的风险。

## 方案选择

采用“单租户私测加固”方案：继续使用一个 Render 服务、一个共享访问码和当前 SQLite/Supabase 结构，不提前建设多租户系统。这个边界足以支撑首轮客户验证，且不会引入账号、权限和账单系统的维护负担。

## 设计

### 1. 移动端布局

- 主容器明确使用 `width: 100%` 和 `min-width: 0`，阻止子元素的最小内容宽度撑开页面。
- 560px 以下压缩顶部栏、快捷问题和输入区间距；快捷问题允许自然换行。
- 输入区宽度始终受视口约束，底部预留 `safe-area-inset-bottom`。
- 长问题、长来源标题和回答正文允许断行，不产生横向滚动。
- 桌面端视觉保持不变，不进行整体重设计。

### 2. 真实访问码验证

- 新增 `POST /api/session/verify`，使用与 `/api/ask` 相同的 `Authorization: Bearer <code>` 校验。
- 登录页只有在该接口返回成功后才写入 `sessionStorage` 并进入主界面。
- 401/403 时保留登录页并显示明确错误，不让客户先进入再失败。
- 从链接读取 `code` 后立即使用 `history.replaceState` 清除地址栏参数，并设置 `Referrer-Policy: no-referrer`，降低访问码进入历史记录和第三方请求来源信息的风险。

### 3. 移除旧码后门

- `check_auth` 只接受当前 `APP_API_KEY`，不再额外硬编码允许 `fb300`。
- 未配置 `APP_API_KEY` 时保留本地开发免验证行为。
- Render 当前配置仍可继续使用 `fb300`；以后在 Render 修改环境变量后，旧码会立即失效。

### 4. 基础调用保护

- 问题去除首尾空白后最多 1000 个字符，超限返回 400 和可理解的提示。
- 按客户端 IP 做内存级滑动窗口限制：每 60 秒最多 12 次需要鉴权的 API 请求。
- 超限返回 429 和 `Retry-After`，前端显示“请求过于频繁，请稍后再试”。
- 该限制用于首轮单实例 Render 私测；未来多实例或正式收费时再迁移到 Supabase/Redis 统一限流。

## 数据流

1. 客户打开 `/app` 或旧的带码链接。
2. 前端清除 URL 中的访问码，并调用 `/api/session/verify`。
3. 验证成功后进入问答界面；失败则停留在登录页。
4. 提问先经过鉴权、频率限制和长度校验，再进入检索与 DeepSeek 生成。
5. 原有 Supabase 提问、反馈和后台分析链路保持不变。

## 错误处理

- 访问码错误：登录页内提示，不进入问答界面。
- 请求过频：保留当前问题文本，提示稍后重试。
- 问题过长：提示压缩问题，不调用模型。
- DeepSeek 超时或异常：继续使用现有错误状态，不暴露服务端密钥或内部异常。

## 验收标准

- 390x844、768x1024、1440x1000 三种视口无横向裁切或不可见按钮。
- 正确访问码可进入，错误访问码不能进入。
- 修改 `APP_API_KEY` 后旧访问码无法继续调用任何受保护接口。
- 第 13 次/分钟请求返回 429，窗口恢复后可继续使用。
- 1001 字问题返回 400，且不会调用检索器或 LLM。
- 现有问答、反馈、后台分析与 59 个既有测试保持通过。

## 暂不包含

- 多客户账号和租户隔离
- 计费、套餐和支付
- 模拟数据看板
- 知识库管理后台
- 分布式限流
